Structured Benchmark Insights & Observations

The Structured Benchmark evaluates the system when the database contains only records that have structured_data fields. All unstructured text is discarded, and only structured metadata is ingested. Below are the key findings:

1. Sub-Second Query Latencies (Ultra-Fast)

Because the database is extremely small (average 205 records vs 1250 in unstructured), search indices fit comfortably in memory, and cosine similarity calculations are minimal. Runtimes average a lightning-fast 0.44 seconds per run, making this setup ideal for real-time systems where database size can be restricted.

2. Complete Absence of Database Pruning

With only 205 records, the database is well below the system's hard cap of 432 active records (defined by max_records_per_scope_kind = 24 across 18 scope-kind combinations). Consequently, 0 records were pruned during ingestion. All stored knowledge is preserved.

3. Global Scope Limitation (Tiers 7-10)

While local, user, and session scopes perform adequately, the retrieval hit rate on global difficulty tiers (Tiers 7-10) drops to 0.00%. This occurs because the expected source documents do not contain structured fields and are completely excluded from the dataset. This highlights a critical limitation: structured-only databases fail if documents are not pre-mapped with schemas.

Unstructured Benchmark Insights & Observations

The Unstructured Benchmark evaluates the system on the full document corpus (~1,250 records), but with all structured_data and data_schema attributes stripped. Retrieval must rely purely on semantic text chunking and embedding vector similarity. Below are the key findings:

1. Aggressive LRU Pruning Bottleneck (High Loss)

The most critical discovery is the impact of the database capacity limit: max_records_per_scope_kind = 24. In unstructured runs, the database receives 1250 records. Because the limit allows only 432 active records across scopes, 818 records (65.4%) are pruned on import. Older records are constantly evicted. Since the expected files are frequently pruned before queries are executed, the retrieval hit rate is artificially capped at a low 19.04%.

2. Massively Slow Runtimes

Runtimes average 1,317.81 seconds (~22 minutes) per run, representing a total wall time of over 55 hours. This occurs because the benchmark setup instantiates a brand new database instance, reads the 1250-record JSONL, splits the text into chunks, and computes ONNX embeddings from scratch for every single run of the 150 runs. This massive embedding computation overhead dominates execution times.

3. Global Scope Retrieval

Unlike the structured variant, unstructured runs maintain a non-zero retrieval hit rate on Tiers 7-10 (global scope). Since all records are imported initially, the documents are available for indexing. However, because of the 65.4% pruning rate, many are evicted, limiting the overall hit rate.

Deep-Dive Comparative Analysis

1. The Pruning Bottleneck (Why hit rates are low)

A central finding across both variants is the low retrieval hit rate (~17% to 19%). This is directly caused by the database LRU eviction algorithm. The service limits the number of active records to max_records_per_scope_kind = 24. When a dataset exceeds 432 total records (across scopes and kinds), older records are deleted to make space. In unstructured and default variants, this wipes out 56% to 65% of all imported documents, meaning the system evicts expected source records before the search is executed.

2. Structured vs. Unstructured Tradeoffs

  • Structured Variant suffers zero pruning since it only contains 205 records (well below the 432 limit). However, it completely lacks coverage for global scopes (Tiers 7-10), leading to a 0.0% hit rate on those queries. It is incredibly fast (0.44s) but lacks recall density.
  • Unstructured Variant retains full semantic coverage but is severely bottlenecked by pruning (65.4% evicted) and exhibits catastrophic runtimes (22 minutes per run) due to embedding-generation overhead.
  • Default Baseline was evaluated on an older set of cases with high term overlaps, yielding a high assembly hit rate (59%) but nearly zero retrieval hits (0.86%), which suggests an embedding/indexing configuration mismatch in those earlier cached runs.
Recommendations for Production Configuration
  • Increase Capacity Limit: Set CONTEXT_BUCKET_MAX_RECORDS_PER_SCOPE_KIND to at least 128 or higher for larger document corpora to prevent LRU eviction.
  • Persistent Indexing: Avoid re-chunking and re-embedding files on every query/session instantiation. Implementing a persistent vector store (such as SQLite or FAISS) would reduce query latencies from 22 minutes to under 50ms.
  • Implement Hybrid Search: Leverage structured fields for metadata filtering combined with semantic vectors for text matching to combine structured speed with unstructured recall.
